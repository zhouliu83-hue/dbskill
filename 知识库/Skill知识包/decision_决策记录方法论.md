# 决策系统方法论

这份文档给 `dbs-decision` 提供深度参考。讲清楚这套方法为什么这么设计、解决什么问题。

## 一、要解决的问题

大多数人的重大决策，最后烂在 4 个地方：

1. 当时的想法没留下来，事后靠印象回忆。
2. 事实、阶段判断、待验证假设、结果回填混在一起，越看越乱。
3. 关键信息散在事件文件里，下次回来抓不住当前主线。
4. 同类问题反复出现，从来没炼成可复用的理解。

这套方法把一次次决策、变化、结果回填，沉淀成长期资产。

## 二、核心目标

用户打开一个项目，第一眼看到的应该是：

- 当前阶段
- 当前最强矛盾
- 当前主线
- 最近节点
- 待回填结果
- 暂不处理

所以这套方法优先维护**当前工作态**，不是事件台账。每次进入项目先更新 `我的当前状态.md`，不是先立案。

## 三、为什么是四层而不是六层

之前的版本分了"业务素材 / 业务理解 / 阶段快照 / 开放问题 / 关键决策"五层 + 当前状态，跑了几次发现问题：

- "关键决策"和"开放问题"在实际操作里是同一层语义（都是"未完成、需要跟踪、有答案要回填"），人为拆开会让用户在两个文件夹之间犹豫。
- 名字带"业务"二字限制了通用性。这套方法在关系、健康、职业等任何长期跟踪领域都能用。
- 层数越多，层间规则越难记。

四层版本：

| 层 | 放什么 | 规则 |
|---|---|---|
| `01_事实/` | 发生过什么 | 只追加 |
| `02_规律/` | 看出什么 | 缓慢追加修正 |
| `03_定格/` | 某时整体什么样 | 写完不改 |
| `04_待解/` | 还没想清楚的（含决策事件） | 完成即清 |

「决策事件」作为 `04_待解/` 里的一种文件类型，不独立成层。

## 四、什么信息先去哪里

### 当前状态变化

先写 `我的当前状态.md`。这是默认入口、每次对话最先更新。

### 长期稳定事实

写 `01_事实/`。例如：

- 产品矩阵变了
- 定价口径变了
- 合作分工明确了
- 某个相关方的稳定特征
- 用户长期偏好和边界

### 可复用的理解

写 `02_规律/`。判断标准：未来还会被反复调用、能解释多个事实、对下一步有指导。

### 还没验证的东西

写 `04_待解/开放问题.md` 或 `待验证假设.md`。

### 重大分叉

新建 `04_待解/决策事件_*.md`。只有 5 种情况触发：产品结构变化 / 重大合作变化 / 价格带变化 / 长期策略反转 / 后续必须回填。

## 五、为什么默认不先立案

很多对话不是在做独立决策，是在不断校正当前状态、主线、优先级、矛盾。

每次变化都先立案，项目会变成档案室。

更好的顺序：

```text
新情况
  ↓
先更新当前状态
  ↓
判断属于事实、规律、待解，还是要立案
  ↓
只在必要时立案
```

## 六、规律层为什么允许先命名

商业、关系、健康里很多重要模式，一出现就值得命名，不需要机械等到第 3 次。

可以先用：

- `[AI 暂定概念 YYYY-MM-DD]`
- `[AI 暂定模式 YYYY-MM-DD]`

后续通过结果回填证实或证伪。详见 [decision_概念炼出方法.md](./decision_概念炼出方法.md)。

## 七、结果回填做什么

回填不只是补一句"后来发生了什么"。至少要做 3 件事：

1. 更新对应决策事件文件
2. 把已确认的事实写回 `01_事实/客观事实.md`
3. 必要时修正 `02_规律/` 的概念或模式（用 `[修正 YYYY-MM-DD]` 追加，不重写）

少了这一步，项目停在记录层，攒不出长期理解。

## 八、和 dbs-learning 的关系

dbs-learning 的成功机制有 4 个能借：

1. 有强入口（`我的当前状态.md` 对应学习的"摘要文件"）
2. 每轮输出都服务下一轮
3. 用摘要压缩复杂历史
4. 最近反馈直接驱动下一步

区别：学习工程是线性的，决策工程是多线并行的。dbs-decision 不能照搬章节结构，但要保留强入口。

## 九、一句话收束

这套方法维护的不是决策记录，是：

**一个围绕某个领域现实、会持续更新、能直接指导下一步动作的本地理解工程。**
